## Context

The repo needs a coherent knowledge management architecture: per-user append-only streams (logs, notes) flowing into compiled shared knowledge (wiki, research). This follows the twyne-root model where no two users write to the same file, and compiled knowledge is the shared artifact.

### B. Write `notes/README.md`

```markdown
# Notes — Per-User Convention

Each team member has a personal directory: `notes/<username>/`

## Structure

- `plans/` — Active plans and design documents
- `research/` — Research observations and notes
- `handoffs/` — Work handoff documents (active/completed/archived)

## Rules

1. Only write to YOUR directory (`notes/<your-username>/`)
2. Never edit another user's files
3. Handoff INDEX.md is auto-generated — don't edit manually
4. Content here is committed (unlike `local/` which is gitignored)

## Handoff Aggregation

`notes/handoffs/INDEX.md` is regenerated by scanning all
`notes/*/handoffs/active/*.md` files. Run `/project-wiki compile`
or the session-end hook triggers aggregation automatically.
```

### E. Create `research-intake` skill

New skill at `agents/skills/research-intake/SKILL.md`:

```markdown
---
name: research-intake
description: Ingest external sources into structured knowledge base
---

# Research Intake

## Trigger
`/research-intake <url1> [url2] ...` or when user provides research material

## Workflow

1. **Fetch**: Download/read each provided source
2. **Dedup**: Check `knowledge/research/intake_index.yaml` — skip already-processed sources
3. **Score**: Rate source relevance to project (1-5 scale based on taxonomy match)
4. **Extract**: Produce structured summary in `knowledge/research/deep-dives/`
5. **Index**: Append entry to `knowledge/research/intake_index.yaml`
6. **Categorize**: Map to `knowledge/taxonomy.yaml` categories (suggest new if needed)

## Output Format

Each deep-dive file: `knowledge/research/deep-dives/YYYY-MM-DD-slug.md`

```markdown
# {Title}

**Source**: {url}
**Ingested**: {date}
**Categories**: {taxonomy categories}
**Relevance**: {1-5}

## Key Takeaways
[3-5 bullet points]

## Detailed Notes
[Structured content]

## Action Items
[What this means for the project, if anything]
```

## Dedup Rules
- Same URL → skip (exact match in intake_index.yaml)
- Same content hash → skip (content-level dedup)
- Same topic, different source → process but cross-reference existing entry
```

Also create thin Claude wrapper at `.claude/skills/research-intake/SKILL.md`.

### F. Update session-start hook for stale wiki detection

Add to `scripts/hooks/session-start.sh` (or the consolidated equivalent after Handoff 5):

```bash
# Stale wiki detection
if [ -f "knowledge/.last_compile" ]; then
  last_compile=$(stat -c %Y "knowledge/.last_compile" 2>/dev/null || echo 0)
  newest_source=$(find logs/progress/ notes/ -name '*.md' -newer "knowledge/.last_compile" 2>/dev/null | head -1)
  if [ -n "$newest_source" ]; then
    echo "ℹ Knowledge base may be stale. Run /project-wiki compile to update."
  fi
fi
```

## Risks

- **Wiki compilation concurrency**: Two users compiling simultaneously could conflict. The design mitigates this by making compilation a local operation that writes to shared output — last writer wins. For templates, this is fine (single user at a time).
- **research-intake generalization**: Upstreaming from epyc-root requires stripping domain-specific scoring. Keep the taxonomy-based scoring generic.
