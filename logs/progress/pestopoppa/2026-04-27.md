### Key decisions

- **Adopted portable subset only.** Skipped kernel sandboxing (per-deployment concern) and OS-keychain integration (heterogeneous on Linux). Documented both as out-of-scope in `docs/guides/security.md` so future contributors know the omission is deliberate.
- **`tools.lock` is gitignored, `tools.lock.example` is committed.** Absolute paths vary per machine, so committing the active lock would create constant per-installation churn. The `.example` is the contract; `tools-init.sh` is the explicit re-approval step.
- **`HOOKS.lock` IS committed.** The whole point is that any clone/fork can detect tampering vs. the canonical tree. A gitignored lock would defeat the threat model.
- **Fail-soft default for `hook_resolve_tool`.** Falling back to `command -v` when unpinned avoids breaking hooks if `tools.lock` is stale or missing. Strict mode (`ARCHETYPE_HOOK_TOOLS_STRICT=1`) is the opt-in for hardened deployments / CI.
- **Did not refactor existing `jq`/`git` callsites.** Helper + lock landed as infrastructure; gradual migration of existing callsites (≈30 inline `jq` invocations) is a follow-up, not part of this change.
